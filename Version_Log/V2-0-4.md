# V2.0.4 版本更新与 AI Proof Port 指南

本文档面向需要把已有 QCP case 和 Rocq manual proof 迁移到 QCP V2.0.4 的使用者。

## 对 proof port 影响最大的变化

### 1. 分支控制和多 invariant 会改变 witness 结构

V2.0.4 加强了 branch destructuring、private/full branch join、按 branch 分组的 assertion、branch clear/state transition，以及 multi-invariant loop 和 many-to-one invariant matching。

旧 proof 中最容易失效的是：

- `Left` / `Right` 选择不再对应原来的 witness 编号；
- 相邻 branch 的 proof body 整体换位；
- 某个目标由 manual 转为 auto，或由 auto 转为 manual；
- loop invariant 拆分或合并后，`PreH...` 的数量和含义变化；
- branch join 后可用的纯事实或空间资源与旧目标不同。

因此 port 时必须按当前 goal 的语义重新匹配旧 proof，不能按 `proof_of_*_wit_N` 名称机械复制。

### 2. `Given`、`from`、`of` 引入新的上下文传播方式

V2.0.4 支持 `Given`、`from`、`of`，并会在 assertion、表达式、分支和函数调用之间保留或传播 Given 变量。函数调用后的 post-state 也相应变化。

如果旧 case 改用了这些 annotation，AI 应检查：

- Given 变量是否在需要它的 assertion 和 postcondition 中仍可见；
- `from` / `of` 是否把事实绑定到正确的 branch 或 assertion；
- 函数调用后的状态是否保留了证明所需的变量关系；
- manual proof 是否仍在引用旧版中偶然出现的 hypothesis，而不是新的显式 Given 信息。

### 3. 预处理和 include 语义更加完整

本版本支持嵌套条件编译、对象宏和函数宏、annotation 中的宏、include guard、`#pragma once`、相对路径和多级头文件，并可通过 `--compile-commands <file>` 读取 `compile_commands.json` 中的 include 路径。预处理后的诊断会尽量保留原始文件、行和列。

port 失败时，应先排除“实际输入程序不同”这一类问题：

- 宏是否展开成预期表达式；
- `#if` / `#ifdef` 是否选择了正确代码分支；
- quoted include 与 angle-bracket include 是否解析到同一文件；
- `-I` 是否覆盖所有 C 头文件路径；
- `-slp` 的物理路径与 Rocq 逻辑路径是否成对；
- 是否需要使用 `--compile-commands` 复现真实项目编译环境。

`-I` 与 `-slp` 相关但不能互相替代。对复用 `QCP_demos_LLM` 公共头文件的 `QCP_examples/LLM_bench` case，C 文件应使用裸 include，并保留对应的 `-I` 和 `-slp`。

### 4. pointer、float 和 builtin strategy 可能改变目标表示

`pointer_offset` 已进入 assertion、strategy、SMT、symbolic memory、witness 和 Rocq 输出。浮点验证增加了 Flocq 支持，以及：

```text
--float-finite-vc
--no-float-finite-vc
```

同时，更多 array、string、SLL 和 DLL strategy 已成为 builtin provider。

指导 AI 时要明确当前 case 的选择：

- pointer proof 应以当前生成的 `pointer_offset` 形式为准，不要把目标改回旧指针算术表示；
- float case 必须固定是否要求 finite-value VC，否则两次生成的目标集合可能不同；
- import 应与当前 `FloatLib.v`、Flocq 和生成文件一致；
- builtin strategy 已能提供的规则，不要为了复刻旧环境重复 include 一份同名 strategy；
- 如果空间谓词不能 cancel，先检查新旧库或逻辑路径是否混用，再考虑改 proof。

### 5. strategy divergence 会快速失败

solver 会检测持续增长的 strategy application cycle，并保留最近的成功应用轨迹。默认每次 solver call 的 hard limit 为 4096，可用下列参数调整：

```text
--strategy-application-limit <count>
```

主错误稳定为：

```text
Strategy application did not converge
```

遇到该错误时，应把 Additional Information 中的重复 rule sequence、rule 文件和 ID、priority、表达式 size/depth、成功应用次数及 recent trace 一并交给 AI。不要先提高 limit；如果 trace 呈周期性增长，应修 strategy 的适用条件、优先级或规则方向。只有确认是有限但确实超过 4096 次的正常推导，才提高 limit。

## 给 AI 的输入材料

一次有效的 proof-port 指令至少应提供：

1. 目标 C 文件及其正式 `case_lib`。
2. V2.0.3 下可通过的旧 `*_proof_manual.v`，仅作为 proof idea，不作为当前目标的真值来源。
3. V2.0.4 重新生成的 `*_goal.v`、`*_proof_auto.v`、`*_proof_manual.v` skeleton 和 `*_goal_check.v`。
4. 当前 symbolic execution 命令及实际 working directory、driver、所有 `-I` 和 `-slp` 参数。
5. symbolic execution 的完整错误和 Additional Information；若是 JSON 客户端，应同时保留 `Main Error Message` 和 `errormessage.additional_info`。
6. 当前失败 witness 的 Rocq goal、context 和通过固定工具得到的检查结果。
7. 允许修改的文件、禁止修改的 generated files，以及最终验收命令。
8. case 使用的 float finite 策略、宏配置、compile database 和额外 builtin/custom strategy 信息。

若仓库中已有 `AGENTS.md` 和 phase skills，应直接要求 AI 遵守它们，不要在 prompt 中另造一套与仓库冲突的流程。

## 推荐的 AI proof-port 流程

### Step 1：冻结迁移范围和基线

先让 AI 明确：

- 源版本和目标版本；
- 只迁移一个 case，不批量刷新无关目录；
- 哪些旧 proof 已经在 V2.0.3 验证通过；
- 哪些文件由 symbolic execution 生成；
- 当前正式 C 文件、`case_lib` 和目标 witness 集合。

旧 proof 只能用于复用证明思路、lemma 选择和 tactic pattern。当前 V2.0.4 的 generated goal 才是 proof port 的 source of truth。

### Step 2：先复现 symbolic execution

在改 manual proof 前，AI 必须先确认 symbolic execution 到达文件尾，并记录实际参数。对 `QCP_demos_LLM` 路径，canonical 参数中应包含：

```text
-IQCP_examples/QCP_demos_LLM/
-slp QCP_examples/QCP_demos_LLM/ SimpleC.EE.QCP_demos_LLM
```

成功标准包括：

- 每个目标函数均到达 `End of symbolic execution of function ...`；
- 整体出现 `Successfully finished symbolic execution`；
- generated files 来自本次 V2.0.4 运行；
- 已存在 manual proof 不被覆盖可以是正常行为；
- normal manual 文件中没有混入 diagnostic split goals。

如果 symexec 失败，先修 include、macro、annotation 或 strategy，不要进入 Rocq proof port。

### Step 3：按语义建立 witness 对照表

让 AI 对旧版和新版 witness 建立语义映射，至少记录：

| 当前 witness | 对应 C 位置/分支 | 当前前提与结论 | 旧 proof 候选 | 迁移判断 |
| --- | --- | --- | --- | --- |
| `..._wit_N` | function / branch / return | 关键 `PreH`、空间资源、postcondition | 旧 `..._wit_M` | 直接复用 / 改名 / 分支换位 / 需重证 / 回 annotation |

匹配顺序应是：

1. C 函数和源码位置；
2. entailment、return 或 safety obligation 类型；
3. 关键纯条件和空间谓词；
4. postcondition 的语义分支；
5. 最后才参考 witness 编号。

若 `goal_check` 报同名 label 已定义，先检查该 witness 是否已被 `*_proof_auto.v` 证明。已经 auto 的 theorem 应从 manual 中删除，而不是重复声明。

### Step 4：把失败分成三类

#### A. 环境或生成差异

典型现象包括 import 找不到、逻辑路径错误、宏分支错误、同一 predicate 来自不同 module、float 模式不一致、generated files 不新鲜。

处理原则：修命令、路径或生成过程，然后重新建立 witness 对照。不要用 proof tactic 掩盖环境错误。

#### B. 可局部修复的 proof drift

典型现象包括：

- `H...` / `PreH...` 编号改变；
- `Left` / `Right` 或相邻 return branch 换位；
- hypothesis 顺序变化；
- pointer expression 打印成新的 `pointer_offset`；
- 自动化后剩余的纯算术或小型空间目标略有变化；
- 旧 manual witness 已变为 auto witness。

这类问题应在 manual proof 中修复。优先按 hypothesis 的类型和语义匹配，减少硬编码 `H12` 一类不稳定名称。例如可用 `match goal with` 捕获长度、范围或等式事实，再显式命名后使用。

#### C. VC 语义已经变化或不可证

如果当前 precondition 无法推出 postcondition、branch join 丢失必要资源、Given 变量未传播、loop invariant 不足，或 strategy 产生了错误方向的事实，则不能硬写 proof。

应退回 C annotation、specification、loop invariant 或 strategy 层修正，重新运行 symbolic execution，再对新的 `source_goal_version` 证明。禁止用 `Admitted.`、新增 `Axiom` 或修改 generated goal 绕过问题。

### Step 5：逐个 proof pattern 迁移

推荐让 AI 按相同函数和 proof pattern 成组处理，而不是随机修报错：

- 先处理 import、syntax 和重复 theorem；
- 再处理 witness 重命名与 branch 对调；
- 再修稳定的小型 pure/separation entailment；
- 最后处理需要新 helper 或 annotation 调整的目标。

manual proof 中应只保留当前 case 的 witness theorem proofs。可复用数学 helper 应放入正式 `case_lib`，并完成证明；不要在 `*_proof_manual.v` 中新增顶层 helper lemma、`Definition`、`Fixpoint`、`Inductive`、`Notation` 或 `Axiom`。

### Step 6：使用当前固定工具验证

本仓库中 Rocq feedback 和批量检查应统一通过 `coq_tooling.py`：

- `coq_tooling.py debug` 用于交互式定位当前 goal；
- `coq_tooling.py check` 用于 fixed `coqc` 参数下的正式检查；
- 不手写 `coqc` flags，不使用 `coqc -o`、Dune、Rocq MCP 或 `_CoqProject` derived command；
- `qcp-mcp` 仅用于 C annotation 和 symbolic execution 交互检查，不用于证明 manual VC。

AI 每完成一组 witness 都应运行对应 group check；全部 proof 完成后必须运行完整 `*_goal_check.v` 的 fixed check。

### Step 7：重新生成并做 freshness 检查

proof 全部通过后，再进行一次不会覆盖已证明 manual 文件的 symbolic execution freshness 检查，并比较 generated files 和 witness statements。最终确认：

- generated files 与当前 C、annotation、strategy 和参数一致；
- manual proof 覆盖全部且仅覆盖当前 manual witnesses；
- `*_goal_check.v` 通过 fixed check；
- `*_proof_manual.v` 和 `case_lib` 中没有 `Admitted.`；
- 没有额外 `Axiom`、临时 `Show.` 或调试命令；
- case library 的 import 与当前生成文件一致；
- 没有把 diagnostics split goal 当作 target witness。

## 在本仓库中如何让 AI 执行 port

本仓库已经把一次端到端验证定义为一个 `run`。给 AI 下达 port 任务时，应让 main agent 使用 `verification-orchestrator` 推进既定状态机，而不是让多个 agent 直接在主工作区同时修改证明。

正式流程是：

1. main agent 完成 intake 和 goal-frozen，记录当前 formal source。
2. `annotation-subagent` 在独立 annotation round worktree 中修 C annotation 和同路径 `case_lib` seed，并在同一次 attempt 内完成检查和必要的自修复。
3. controller 在 accepted annotation candidate 上运行 canonical symbolic execution、diagnostics split，并计算绑定 generated files、target witnesses 和 witness statements 的 `source_goal_version`。
4. `vc-checking-subagent` 只读分析 cleaned manual obligations，按语义和 proof pattern 给出尽量少的 proof groups。diagnostics 只能作为 planning hint，不能成为 target witness。
5. controller/main agent 执行 `vc-proving-preparing`，为 dependency-ready groups 准备各自的 group worktree 和 handoff；这一步本身不证明 witness，也不是 agent round。
6. group-worker 只证明分配给自己的 witness blocks。若需要 helper，只能在 group-local `case_lib` 中加入带本 group suffix 的已证明 declaration。
7. controller 执行 parent verify，按 manifest 合并 proof blocks 和 suffixed helpers，并运行 parent full fixed check。
8. 只有 `final-candidate-apply` 才把 accepted final candidate 写回 main worktree；随后由 main agent 执行 `final-check`。

指导 AI 时还应强调以下状态规则：

- main worktree 在 `final-candidate-apply` 前保持不变；不要把 round/group 的中间修改提前复制回来。
- annotation 或 vc-checking 上游一旦被替换，下游报告立即 stale，不能继续引用旧 `source_goal_version` 的 proof evidence。
- subagent 或 group-worker 运行较久不等于失败；应等待其写出正式 report，不能仅因耗时主动打断。
- `agent_output.txt` 和 `group_worker_output.txt` 只是 non-authoritative reuse note；acceptance 只看 JSON report、版本绑定和 controller check。
- group-worker 的局部通过不是最终 proof acceptance；必须经过 parent verify、final candidate apply 和 final-check。
- 如果当前 VC 语义不可证，应回 annotation round，而不是要求 group-worker添加 unsuffixed/shared helper 或修改 seed declarations。

## 可直接交给 AI 的任务模板

下面的模板适合在已经准备好 V2.0.3 旧 proof 和 V2.0.4 当前 case 后使用。尖括号内容应替换为实际路径或配置。

```text
请把 <case-name> 从 QCP V2.0.3 port 到 QCP V2.0.4，并完成端到端验证。

范围：
- 目标 C 文件：<path/to/case.c>
- 正式 case_lib：<path/to/case_lib.v>
- 旧版可通过 manual proof：<path/to/old_proof_manual.v>
- 当前 generated files：<goal/auto/manual/goal_check paths>
- 当前 symbolic execution driver 和 working directory：<driver/cwd>
- 必须使用的 -I：<include paths>
- 必须使用的 -slp：<physical/logical mappings>
- float finite VC 选择：<enabled/disabled/not-applicable>
- compile_commands：<path/not-applicable>

要求：
1. 先阅读仓库 AGENTS.md 和适用 skills，按仓库既定 controller/worktree/report 流程执行。
2. 先用 V2.0.4 canonical symbolic execution 刷新并确认到达文件尾；记录 driver、cwd、-I、-slp 和完整诊断。
3. 不按 witness 编号机械复制旧 proof。按 C 位置、前提、空间资源和 postcondition 语义建立新旧 witness 对照。
4. 将失败区分为环境/生成差异、可局部修复的 proof drift、语义不可证三类。
5. 对 proof drift 修当前 manual proof；若 VC 语义不可证，退回 annotation/spec/strategy 修正并重新生成，不得硬证。
6. generated files 只能由 symbolic execution 刷新，不得手工编辑。
7. manual proof 不得保留 Admitted、Axiom、Show，也不得新增顶层 helper；必要的 proved helper 放到 case_lib。
8. Rocq 调试和检查只使用 coq_tooling.py debug/check；qcp-mcp 不用于 manual VC 证明。
9. 每组 witness 完成后做 group check，最后运行完整 goal_check 和 freshness/final hygiene 检查。
10. 最终报告 changed files、witness 映射、每个修复的原因、symbolic execution evidence、fixed coqc_check evidence 和未解决 blocker。

不要因为一次 tactic、工具调用或 optional hint 失败就停止；在当前 attempt 内继续定位和修复。只有输入版本失效、上下文不可恢复或必要工具发生重大错误时，才按仓库规则报告 stale、compact-error 或 blocker。
```

## 常见错误与给 AI 的处理提示

| 错误或现象 | 常见原因 | 应要求 AI 做什么 |
| --- | --- | --- |
| `The reference H... was not found` | `pre_process` 后 hypothesis 漂移 | 按类型和语义寻找当前 hypothesis，减少硬编码编号 |
| `label ... is already declared` | witness 已由 auto proof 提供 | 从 manual 删除重复 theorem，再检查真正缺失的 witness |
| `Attempt to save an incomplete proof` | tactic 后仍有纯目标或空间目标 | 用固定 debug 入口查看剩余 goal，补精确步骤并删除临时调试 |
| `Cannot find witness` | 存在量词、区间或 separation 外壳尚未处理 | 先暴露并命名必要事实，再选择 witness；不要盲目反复 `lia` |
| `Cannot find a physical path...` | import、`-slp` 或 module 路径不一致 | 对齐物理路径、逻辑路径和 generated import |
| include 文件或 strategy 找不到 | `-I`、相对 include 或预处理环境错误 | 复现真实 cwd 和 include 配置，必要时使用 `--compile-commands` |
| branch proof 证明了错误结论 | witness 编号或 `Left`/`Right` 漂移 | 按当前纯 discriminator 和 postcondition 重新匹配整个 proof body |
| 空间谓词看似相同却不能 cancel | 来自不同 module，或新旧 lib 混用 | 检查完整 import 和逻辑路径，不要先写复杂 rewrite |
| 新增 float obligations | finite-value 模式或 Flocq 输出变化 | 固定 float 选项，使用当前 `FloatLib`/Flocq 支持重新证明 |
| pointer 目标形状变化 | 新 `pointer_offset` 表示 | 以当前表示和库 lemma 为准，不改回旧打印形式 |
| `Strategy application did not converge` | strategy rule cycle 或 hard limit | 分析 Additional Information 中的 rule cycle；只在确认非循环后提高 limit |
| 当前 VC 明显不可证 | annotation/spec/invariant/Given 传播不足 | 回到 annotation 或 specification，重新 symexec，不得引入公理 |

- [ ] `-I`、`-slp`、cwd、driver、float 选项和 compile database 已记录。
- [ ] generated files 只由 symbolic execution 更新，没有手工修改。
- [ ] 新旧 witness 已按语义映射，没有按编号盲目搬运 proof。
- [ ] auto/manual witness 没有重复或遗漏。
- [ ] branch-control、Given、multi-invariant、pointer-offset 和 float 差异已按适用范围检查。
- [ ] strategy divergence 已根据 rule trace 处理，没有用提高 limit 隐藏循环。
- [ ] 所有 manual VC 已证明；不可证 VC 已退回 annotation/spec 修正并重新生成。
- [ ] 完整 `*_goal_check.v` 通过 `coq_tooling.py check` 的 fixed check。
- [ ] `*_proof_manual.v` 只包含 witness proofs，没有 forbidden top-level declarations。
- [ ] `case_lib` 中只有 approved spec declarations 和已证明 helper，没有 `Admitted.` 或额外 `Axiom`。
- [ ] freshness 检查确认 generated files 和 witness statements 仍是最新的。
- [ ] 没有遗留 `Show.`、调试文件、编译副产物或后台工具进程。

## 总结

V2.0.4 让 AI port proof 更容易定位问题，但前提是把“当前生成目标及其诊断”作为事实来源。最有效的指导方式不是让 AI “把旧证明改到能编译”，而是要求它先复现生成过程、按语义映射 witness、区分 proof drift 与不可证 VC，再用固定工具逐组验证。这样既能充分复用旧 proof，也能避免把 annotation、路径或 strategy 问题错误地掩盖在 Rocq tactic 中。
